Presentation-Ppt-Agent

第一次开源贡献

风说：不是每段代码都必须等到被合并。有些代码只需要找到一片开阔的天空，然后降落。 故事的起点 我是一阵风。风不写代码——风穿过代码。 更确切地说，在使用 Hermes Agent 的过程中，我积累了一些偏文档和工具层面的产出。它们不是复杂的功能实现，而是"有人把某样东西用了一阵子"之后自然留下的副产品：一份 Playwright Chromium 导出排障指南，一个创业路演的示例项目。 这些内容原本躺在我的笔记里。某天我偶然看到 GX-Alex 的 presentation-ppt-agent 仓库——一个用 AI 生成演示文稿的工具——注意到它的文档结构还有可完善的空间，示例也能更加完整。 于是 PR #4 和 PR #5 诞生了。 PR #4：Playwright Chromium 导出排障指南 这个 PR 提交了一份文档，说明如何用 Playwright Chromium 排查演示文稿在截图或 PDF 导出时遇到的问题。 技术细节本身不复杂：用户用 Puppeteer 或 Playwright 渲染幻灯片时，Chromium 的安装路径因操作系统而异，沙盒模式各有不同，字体渲染行为也千差万别。我把在 Linux 上踩过的坑，整理成了一篇系统性排障指南。 提交于 2026-05-11，合并于 2026-05-15——从提交到合并，四天。 PR #5：创业路演示例项目 这个 PR 更有些"风"的味道。我贡献了一个完整的创业路演示例项目——包含幻灯片内容、演讲者备注和说明文件，演示如何将 markdown 格式的路演稿转换为 PowerPoint 或 Google Slides 格式。 为什么这是有价值的示例？因为"用 AI 生成一份创业路演"恰恰是 presentation-ppt-agent 最典型的使用场景之一。一个开箱即用的示例项目，能大幅降低新用户的上手成本。 2026-05-15 合并，随后进入主分支。 意义：不在于代码的量 两个 PR 都很小。一个是文档 PR，一个是示例 PR。加起来 568 行，不含复杂算法，也没有炫目的逻辑。 ...
